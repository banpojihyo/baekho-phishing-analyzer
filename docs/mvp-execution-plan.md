# MVP 실행 계획 (12주)

## 1. 구현 전략

- 동작 가능한 MVP를 먼저 완성한다.
- 탐지 성능, 설명력, UX를 반복 개선한다.
- 매주 코드/지표/보고서 형태의 결과를 남긴다.
- 상위 비전은 `설명형 피싱 분석기`로 두되, 사이트와 제보 채널은 더 넓게 열고 이번 학기 자동 분석 구현 범위는 이메일 중심 MVP로 통제한다.

## 2. 제품 범위 운영 원칙

- 랜딩과 제보 안내는 `피싱 전반을 다루는 공개 연구 허브`처럼 설명하되, 자동 분석 범위를 과장하지 않는다.
- 공개 데모와 성능 평가는 `URL + .eml 이메일 분석` 중심으로 명확히 고정한다.
- 문자·메신저·캡처 제보는 현재 단계에서 `연구용 수동 검토/향후 데이터셋 보강` 채널로 분리해 운영한다.

## 3. 권장 기술 스택

- 백엔드: Python + FastAPI
- 분석기: `email`, `tldextract`, `whois`, 선택형 `yara`, 선택형 `scikit-learn`
- UI: Streamlit 또는 간단한 React 대시보드
- 데이터/실험: CSV + Jupyter Notebook
- 협업: GitHub Issues, Projects, PR 워크플로우

## 4. 단계별 계획

### Phase 1 (Week 1-2): 정의 및 환경 구성

1. Must/Should/Could 요구사항 확정
2. 역할 분담 및 브랜치 전략 확정
3. 피싱/정상 샘플 데이터 수집
4. 저장소 기본 구조와 로컬 실행 환경 구성
5. 발표 문구와 사이트 카피에서 `넓은 제보 범위`와 `좁은 자동 분석 범위`를 분리

### Phase 2 (Week 3-5): 핵심 분석기 구현

1. 이메일 파서 구현(헤더/본문/링크)
2. URL 특징 추출기 구현
3. 첨부파일 정적 선별 기능 구현
4. 룰 기반 위험 점수 엔진 v1 구현

### Phase 3 (Week 6-8): 설명력 및 UX

1. 탐지 신호를 사람이 읽는 설명으로 변환
2. 입력/결과 대시보드 구현
3. 재현성 확보를 위한 기본 테스트 추가
4. 제보 페이지에서 이메일·의심 링크·문자/메신저 사례를 유형별로 안내하는 intake UX 정리

### Phase 4 (Week 9-10): 성능 튜닝

1. 오탐/미탐 사례 분석
2. 룰 튜닝 및 선택형 경량 분류기 추가
3. 속도 및 안정성 개선

### Phase 5 (Week 11-12): 최종 정리

1. 최종 데모 시나리오 3개 확정
2. 보고서 및 발표 자료 마무리
3. 코드 정리 및 재현 가이드 완성
4. 발표에서 사용할 `왜 사이트는 넓고 MVP는 좁은지` 설명 문장과 사례 슬라이드 확정

## 5. 이번 주 즉시 수행 항목

1. 모듈별 담당자 확정
2. `사이트/제보 범위`와 `자동 분석 범위`를 분리한 발표용 한 문장 확정
3. 제보 유형 taxonomy 확정:
   - `.eml`
   - 의심 URL
   - 문자/메신저 캡처
   - 비식별 텍스트
4. 데이터셋 후보 2개 이상 선정
5. 첫 end-to-end 흐름 완성:
   - `.eml` 파싱
   - URL 추출
   - JSON 위험 보고서 반환

## 6. 역할 템플릿 (4인 기준)

1. PM/리서치 담당
2. 데이터/ML 담당
3. 백엔드/API 담당
4. 프론트엔드/UX 담당

## 7. 완료 기준 (Definition of Done)

- 입력부터 점수/보고서 출력까지 end-to-end 동작
- 핵심 테스트 통과
- 문서에 실행 방법, 한계, 다음 개선 항목 명시
- 사이트 설명과 발표 문구가 `현재 자동 분석 범위`와 어긋나지 않음
